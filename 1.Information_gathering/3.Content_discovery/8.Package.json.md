# package.json.bak Information Disclosure

### Observation
- A backup copy of `package.json` was accessible through the `/ftp/` directory.
- The file disclosed application metadata, technology stack information, dependency lists, version information and runtime requirements.

### Access Method
- Direct access to the backup file returned a `403` response, revealing that only `.md` and `.pdf` files were permitted by the application's
  extension whitelist.
- <img width="600" alt="image" src="https://github.com/user-attachments/assets/4cb2aeed-ad07-4655-b619-34a5b7fb35c0" />
- Appending an allowed extension (`package.json.bak.md`) resulted in a `404` response, confirming that the whitelist check had been bypassed but
  the file did not exist under that name.
- A null byte payload was appended to the filename to bypass the extension validation, but the request returned a 404 response, indicating that
  the application processed the filename differently and the file was not disclosed.
- A double-encoded null byte payload was then appended to the original filename: `http://localhost:3010/ftp/package.json.bak%2500.md`

### Analysis
Exposed package files may reveal:
- Technology stack
- Framework versions
- Security libraries
- Development tooling
- Runtime requirements
- Potentially vulnerable dependencies
This information significantly reduces the effort required to profile the application during reconnaissance.

## Recon Takeaway
Backup files often contain sensitive development information. An exposed `package.json` file can reveal the application's architecture, 
dependencies, and version information, enabling technology fingerprinting, attack surface mapping, and vulnerability research without requiring 
access to source code.
