## suspicious_errors.yml

### Initial Observation
While enumerating files under the `/ftp/` directory, the file `suspicious_errors.yml` was discovered.

### Recon Process
#### Step 1: Direct Access Attempt
Request:
```text
http://localhost:3010/ftp/suspicious_errors.yml
```
- <img width="700" alt="image" src="https://github.com/user-attachments/assets/b2382a22-86a2-4d96-872d-19130b6da8ed" />
- <img width="700" alt="image" src="https://github.com/user-attachments/assets/cda32922-8e34-4308-8895-b9b65a33ab6d" />

Response:
```text
403 Error: Only .md and .pdf files are allowed!
```
This indicated that the application enforces an extension whitelist and only permits access to files ending in `.md` or `.pdf`.

#### Step 2: Testing the Whitelist
To verify the behavior, the filename was modified by appending an allowed extension:
```text
suspicious_errors.yml.md
```
Response:
```text
404 Not Found
```
- <img width="700" alt="image" src="https://github.com/user-attachments/assets/edbb586d-e834-4209-aa4d-d386297a7483" />
This suggested that the whitelist check was bypassed successfully because the request was no longer blocked with a `403` error.
However, the file did not exist under that name, resulting in a `404`.

#### Step 3: Null Byte Injection Attempt
A null byte payload was used to try to satisfy the whitelist while forcing the backend to process the original filename.
Example concept:
```text
suspicious_errors.yml%00.md
```
Response:
```text
400 Bad Request
```
- <img width="700" alt="image" src="https://github.com/user-attachments/assets/f75bc8c8-ce10-452b-a73c-c5a36e1853b5" />
The application rejected the request.

#### Step 4: Double-Encoded Null Byte
- A double-encoded null byte payload was then tested.
- The request passed the extension validation and allowed access to the original file content.

### Validation
One of the error messages contained within the file is:
```text
Only *.md files are allowed
```
This corresponds directly to the error message returned when attempting to access the file without an allowed extension.

### Why Interesting?
The file reveals:
- Application validation logic
- Security control behavior
- Error handling rules
- Indicators used to identify suspicious or unauthorized activity
- Internal application logic that is normally not intended for public disclosure

### Recon Takeaway
The discovery process demonstrated how extension-based access controls were implemented and how alternative encoding techniques affected request processing.
Publicly accessible configuration or detection files can provide valuable insight into application behavior, validation mechanisms, and security controls. 
Such information helps build a more complete understanding of the target during reconnaissance and security testing.
